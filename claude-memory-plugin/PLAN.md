# Plan: Ralph-Loop Spec for Always-On Memory Claude Code Plugin

## Context

Developers using Claude Code lose all session context between conversations - architectural decisions, bug patterns, failed approaches, conventions, and debugging insights all vanish. Auto-memory (MEMORY.md) captures only a fraction, is limited to 200 lines, and Claude controls what gets saved. This costs developers 15-30 minutes per context-restore incident and hours per week.

The always-on-memory-agent repo demonstrates a powerful concept: continuous background memory consolidation (like brain sleep cycles) using a lightweight LLM. We want to adapt this concept into a Claude Code plugin that:

1. **Records** session activity automatically via hooks (no manual inbox drops)
2. **Extracts** useful observations from sessions using an Observer pattern
3. **Consolidates** memories in the background between sessions
4. **Injects** the most relevant memories into new sessions automatically

## What We're Building

A ralph-loop project scaffold that defines specs for building **"claude-memory"** - a Claude Code plugin providing always-on memory for dev workflows. The plugin will be installable via one command and require zero configuration.

### 2. Spec Definitions (one per topic of concern)

#### `specs/session-recording.md` - Automatic Session Capture
- **JTBD**: Capture everything useful from a coding session without developer intervention
- **Mechanism**: Claude Code hooks (PostToolUse, PreCompact, Stop, SessionEnd)
- **What to capture**:
  - Tool calls and outcomes (file edits, bash commands, search results)
  - Architectural decisions expressed in conversation
  - Errors encountered and how they were resolved
  - Files touched and relationships discovered
- **Data flow**: Hooks → append events to local JSONL log per session
- **Key constraint**: Hooks must be fast (<500ms) - async background writes only, never block Claude
- **Transcript access**: Every hook receives `transcript_path` pointing to session JSONL

#### `specs/memory-extraction.md` - Observer Pattern for Distilling Sessions
- **JTBD**: Transform raw session recordings into structured, useful observations
- **Approach**: Mastra-inspired Observer pattern - distillation, not summarization
- **When it runs**: Triggered by SessionEnd hook as a background process
- **What it extracts** (priority-tagged):
  - Architectural decisions + rationale (highest priority)
  - Bug fixes with root causes and solutions
  - File relationships and cross-file dependencies
  - Failed approaches and dead ends (what NOT to do)
  - Code patterns/conventions established
  - Dependency quirks and workarounds
  - Test strategies that worked
- **Output format**: Priority-tagged observations with exact details (file paths, error messages, decisions - never approximate)
- **LLM**: Configurable via env var (default: Claude Haiku). Supports Anthropic, Google, OpenAI, or local models via thin provider abstraction

#### `specs/memory-storage.md` - SQLite Memory Database
- **JTBD**: Store, query, and manage memories efficiently with zero external dependencies
- **Storage**: SQLite database at `~/.claude/projects/<project>/claude-memory.db`
- **Schema** (inspired by always-on-memory-agent):
  - `observations` table: id, session_id, source_file, content, entities (JSON), topics (JSON), priority, created_at, consolidated
  - `consolidations` table: id, source_ids (JSON), summary, insight, connections (JSON), created_at
  - `sessions` table: id, session_id, branch, started_at, ended_at, summary
- **Queries**: Full-text search on content, topic/entity filtering, recency weighting
- **No vector DB**: Keep it simple like the original agent - pure LLM-based retrieval
- **Cleanup**: Auto-prune observations older than 30 days with low relevance scores

#### `specs/memory-consolidation.md` - Background Sleep-Cycle Processing
- **JTBD**: Connect, compress, and improve memories between sessions (like brain during sleep)
- **When it runs**: After session ends (background), configurable interval for continuous mode
- **What it does** (adapted from always-on-memory-agent):
  1. Read unconsolidated observations (most recent, max 20)
  2. Find connections between them (cross-session patterns)
  3. Generate synthesized insights across observations
  4. Detect and resolve contradictions with current codebase state
  5. Mark source observations as consolidated
- **Connection format**: `{from_id, to_id, relationship}` linking related observations
- **Intelligent forgetting**: Decay scores over time, prune low-value observations
- **LLM**: Configurable (same provider abstraction as extraction)

#### `specs/context-injection.md` - Smart Memory Injection at Session Start
- **JTBD**: Automatically inject the most relevant memories into new sessions so devs never lose context
- **Mechanism**: SessionStart hook queries memory DB and returns relevant memories via `additionalContext`
- **Relevance scoring**: Combine recency, priority, topic match (based on git branch, recent files, working directory)
- **Token budget**: Cap injected context at ~4K tokens (~50 lines of observations) to stay lean
- **Format**: Concise bullet points organized by topic, not chronological dumps
- **Progressive disclosure**: Tell Claude WHERE to find deeper memories rather than injecting everything
- **SubagentStart hook**: Optionally inject subset of memories into subagents too
- **Fallback**: If memory DB is empty or query fails, degrade gracefully (no injection, no error)

#### `specs/plugin-packaging.md` - Plugin Structure and Distribution
- **JTBD**: One-command install, zero-config, works immediately
- **Plugin structure**:
  ```
  claude-memory/
  ├── .claude-plugin/
  │   └── plugin.json
  ├── hooks/
  │   ├── hooks.json
  │   ├── session-start.sh      # Inject memories
  │   ├── post-tool-use.sh      # Record events (async)
  │   ├── pre-compact.sh        # Backup before compaction
  │   ├── session-end.sh        # Trigger extraction + consolidation
  │   └── stop.sh               # Optional: trigger mid-session extraction
  ├── scripts/
  │   ├── extract.py            # Observer: distill sessions into observations
  │   ├── consolidate.py        # Consolidator: connect and compress
  │   ├── query.py              # Query memory DB for relevant context
  │   └── setup.sh              # First-run setup (create DB, install deps)
  ├── skills/
  │   └── memory-query/
  │       └── SKILL.md          # /memory-query slash command for manual queries
  ├── README.md
  └── LICENSE
  ```
- **Installation**: `claude plugin install claude-memory` (via marketplace) or `claude --plugin-dir ./claude-memory`
- **Dependencies**: Python 3.10+ (for LLM extraction scripts), sqlite3 (built-in)
- **Configuration**: Optional env vars:
  - `CLAUDE_MEMORY_PROVIDER` (default: `anthropic`) - LLM provider: `anthropic`, `google`, `openai`, or `local`
  - `CLAUDE_MEMORY_MODEL` (default: `claude-haiku-4-5`) - model name for the chosen provider
  - `CLAUDE_MEMORY_API_KEY` (default: falls back to `ANTHROPIC_API_KEY` / `GOOGLE_API_KEY` / `OPENAI_API_KEY`)
  - `CLAUDE_MEMORY_MAX_INJECT_TOKENS` (default: 4096) - max tokens to inject at session start
  - `CLAUDE_MEMORY_RETENTION_DAYS` (default: 30) - auto-prune memories older than this
- **First run**: setup.sh creates SQLite DB, validates Python, installs minimal deps

### 4. Key Design Decisions

1. **Python for extraction scripts** (not Node/bash): The extraction and consolidation logic requires LLM calls and SQLite operations - Python with the Anthropic SDK is the most natural fit and mirrors the original agent's approach.

2. **No vector DB**: Following the original agent's philosophy - "just an LLM reading, thinking, and writing." Full-text search + recency + priority scoring is sufficient for project-scoped memories.

3. **Hooks over MCP**: Hooks fire automatically on session events. MCP tools require Claude to decide to call them. For transparent, zero-effort memory capture, hooks are the right mechanism.

4. **Per-project storage**: Memories are scoped to the project (via `~/.claude/projects/<project>/`), matching Claude Code's existing project model. Cross-project patterns are a future enhancement.

5. **Configurable LLM backend**: Default to Claude Haiku for extraction/consolidation, but support Gemini, OpenAI, or local models via `CLAUDE_MEMORY_MODEL` and `CLAUDE_MEMORY_API_KEY` env vars. The original agent uses Gemini Flash-Lite - users can keep that or use whatever they prefer. Use a thin provider abstraction in `src/scripts/llm_provider.py`.

## Research Findings

### Hook System (17 event types available)
- **SessionStart** - Load prior memories
- **PostToolUse** - Capture tool results and file changes (with matcher filtering)
- **PreCompact** - Save context before compaction
- **Stop** - Extract final state, save session summary
- **SessionEnd** - Trigger extraction + consolidation as background process

### Hook I/O Format
- **Input**: JSON on stdin with `session_id`, `transcript_path`, `tool_name`, `tool_input`
- **Output**: JSON on stdout with `decision`, `reason`, `systemMessage`, `additionalContext`
- **Timeout**: Default 10 minutes, but hooks should complete in <500ms for UX

### Transcript Format
- JSONL at `~/.claude/projects/{PROJECT_HASH}/{SESSION_ID}.jsonl`
- Entry types: user, assistant, tool_use, tool_result, thinking, file-history-snapshot
- Full conversation history with timestamps

### Context Injection Points
1. **PreToolUse** `additionalContext` - Dynamic context before tools run
2. **CLAUDE.md** - Static project context (auto-loaded)
3. **Stop** `systemMessage` - Message shown on next iteration
4. **Agent memory frontmatter** - Persistent state (user/project/local scopes)
